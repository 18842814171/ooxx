# 0705 三天验收计划

> 依据仓库现有示例与 `重要原则.txt` 编写。当前不需要视觉模块；数值参数集中写入 `config/mission.yaml`。

---

## 一、问题界定

实车主矛盾并非「看不见障碍」，而是「已知可通行方向，最终未选中」。典型链路：局部规划器在约负七十度方向给出最高分 → 恢复后偏角约束（三十五度）过滤 → 落到零度 → 撞墙。该问题出在局部规划器评分、恢复后约束与引导期出口，**深度相机无法介入**。

深度信息适合作为第三优先级补充，解决泡沫块、相机支架盲区、低矮障碍、雷达漏检等场景，属于锦上添花，非雪中送炭。

完整扫描融合（深度图投影为激光扫描并替换话题）虽可保持状态机接口不变，但需外参标定、光学坐标系对齐、单位与变换方向排查，调试变量与引导期、缝隙模式、局部规划器问题叠加，**三天内风险过高**。本计划采用降级方案：**不替换雷达扫描话题**，仅在距离读取处做保守合并。

---

## 二、总体策略

| 优先级 | 内容 | 改动范围 |
|--------|------|----------|
| 1 | 引导期、缝隙模式、局部规划器实车验证收尾 | 以验证为主，参数微调 |
| 2 | 深度前方距离补充（可选开关） | 仅 `lidar_utils` + 深度读取 |
| 3 | 完整扫描融合 | **本阶段不做**，留作后续扩展 |

架构约束不变：仅状态机做决策，仅当前控制权拥有者发布非零速度（见 `docs/01-architecture.md`）。

## 四、深度最小方案（优先级 2）

### 4.1 数据流

```
深度图 + 相机内参
    ↓ 取图像左/中/右三列中值距离（米）
模块缓存：center_depth / left_depth / right_depth
    ↓
lidar_utils 三处出口对雷达距离做 min 合并
    ↓
clearance_profile / is_boundary / front_sector_clear
```

状态机、局部规划器、前沿导航器**不改接口**；`/scan` 话题不变。

### 4.2 伪代码

```python
# depth_front_reader.py（新建）或并入 ooxx_node 订阅回调
center_mm = depth_at_pixel(depth, u_center, v_center)
center_m = center_mm * 0.001
set_depth_supplement(center=center_m, left=..., right=...)

# lidar_utils.py — 唯一逻辑改动点
def _merge_range(lidar_r, depth_r):
    if depth_r is None: return lidar_r
    if lidar_r 无效: return depth_r
    return min(lidar_r, depth_r)

# clearance_profile 内：
center = _merge_range(sector_min(...), _supplement.center)
```

配置段 `sensor_fusion`（写入 `mission.yaml`）：`enabled`、`depth_cam_name`、`block_dist`（默认 0.35）、三列像素索引、`max_depth_m`。默认 `enabled: false`，与优先级 1 验证解耦。

## 五、三日分工

### 第 1 天：引导期与缝隙模式实车验证

1. 按 `plan.md` P3b 验收项跑满场：日志出现且仅一次 `Bootstrap exit`；`PassageMode auto-arm` 与 `done`；恢复后 `缝隙规划` 可选中约负七十度方向。
2. 重点核对 `recovery_replan_max_angle_deg: 35.0` 是否误杀大角度候选；必要时仅调 `recovery_passage_replan_angle_deg: 70.0` 与 `recovery_passage_replan_rounds`。
3. 记录边界次数、覆盖率、恢复耗时，作为基线。

**验收**：开阔区以 `drive`/`align_arc` 为主；边界事件不增；已知出口场景能选中非零度目标。

### 第 2 天：局部规划器收尾 + 深度链路（若第 1 天通过）

1. 上午：Failure Memory、贴墙加罚、墙角破格候选（`plan.md` P2b）回归；确认 `-70°` 可选中且不再死循环回零度。
2. 下午：启动 `depth_cam.launch`；实现三列深度读取与 `set_depth_supplement`；`fusion_enabled: false` 下确认主栈无回归。
3. 低矮挡板场景：`enabled: true`，验证 `center` 日志小于纯雷达值，前沿导航器提前停车。

**验收**：深度关闭时行为与第 1 天一致；深度开启时仅影响正前方挡障，不改变规划选角。

### 第 3 天：参数收敛与文档

1. 微调 `block_dist` 与 `boundary_dist: 0.30` 关系，避免误触发 hold。
2. 连续跑场不少于十五分钟；更新 `使用方法.txt` 分步启动与话题检查项。
3. 删除临时调试脚本；不保留并行旧路径。

**验收**：跑场无扫描超时；引导期/缝隙/规划问题已关闭；深度为可选增强。

## 六、风险与对策

| 风险 | 对策 |
|------|------|
| 深度与规划问题混淆 | 前两日 `fusion_enabled: false`；第三日单独开深度对比 |
| 仅改分类不改挡障 | 合并写入 `clearance_profile` 的 `center`，而非只改 `interpret_obstacle` |
| 恢复后偏角过滤过严 | 优先调 `recovery_passage_replan_angle_deg`，不动扫描融合 |
| 深度噪声误停 | 列中值 + 最小有效深度阈值（配置项） |

## 八、结论

三天验收以 **引导期 / 缝隙模式 / 局部规划器** 实车收尾为第一优先级；深度相机以 **前方三距离保守合并** 为可选第二优先级，改动限于 `lidar_utils` 一处；完整扫描融合推迟至验收后。该排序与当前失败模式一致，调试面可控，符合「上层决策修改最小」原则。
